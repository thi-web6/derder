Cara Mengatur SSH di Fedora dengan Autentikasi Berbasis Kunci

Dokumentasi ini menyediakan panduan komprehensif untuk mengatur layanan Secure Shell (SSH) pada sistem operasi Fedora, dengan fokus pada autentikasi berbasis kunci publik-privat. Tujuannya adalah memungkinkan akses jarak jauh yang aman tanpa ketergantungan pada kata sandi, sambil memastikan pengguna lain hanya dapat mengakses server dengan kunci publik yang valid.

Instalasi dan Aktivasi Layanan SSH

Langkah awal dalam pengaturan SSH adalah memastikan layanan OpenSSH Server terpasang dan aktif pada sistem Fedora. OpenSSH menyediakan protokol komunikasi jarak jauh yang aman dan andal.

Langkah-langkah:

  1. Memeriksa dan menginstal OpenSSH Server:

    Jalankan perintah berikut untuk memverifikasi keberadaan paket openssh-server. Jika belum terpasang, perintah ini akan menginstalnya secara otomatis.

    sudo dnf install openssh-server

    Catatan: Jika output menunjukkan bahwa paket sudah terpasang (misalnya, openssh-server-9.9p1-11.fc42.x86_64), lanjutkan ke langkah berikutnya.

  2. Mengaktifkan dan menjalankan layanan SSH:

    Perintah berikut memastikan layanan sshd aktif saat sistem dinyalakan dan berjalan saat ini.

    sudo systemctl enable sshd
sudo systemctl start sshd
  3. Memverifikasi status layanan:

    Periksa apakah layanan sshd berjalan dengan benar menggunakan perintah berikut.

    sudo systemctl status sshd

    Hasil yang diharapkan: Output harus menunjukkan status active (running). Jika layanan tidak aktif, periksa log dengan sudo journalctl -u sshd untuk mendiagnosis masalah.

Potensi masalah: Jika instalasi gagal, pastikan koneksi internet stabil dan repositori Fedora diatur dengan benar. Gunakan sudo dnf repolist untuk memeriksa repositori aktif.

Pembuatan Pasangan Kunci SSH

Pasangan kunci SSH, yang terdiri dari kunci publik dan privat, memungkinkan autentikasi aman tanpa kata sandi. Proses ini dilakukan pada komputer klien yang akan mengakses server.

Langkah-langkah:

  1. Menghasilkan kunci SSH:

    Gunakan perintah berikut untuk membuat kunci RSA dengan panjang 4096-bit, yang memberikan tingkat keamanan tinggi.

    ssh-keygen -t rsa -b 4096
  2. Menentukan lokasi penyimpanan kunci:

    Ketika diminta lokasi berkas, tekan Enter untuk menggunakan lokasi default (~/.ssh/id_rsa untuk kunci privat dan ~/.ssh/id_rsa.pub untuk kunci publik). Jika berkas sudah ada, ketik y untuk menimpa.

  3. Mengatur passphrase:

    Saat diminta passphrase, Anda memiliki dua opsi:

    • Login tanpa kata sandi: Tekan Enter dua kali (untuk passphrase dan konfirmasi) untuk mengosongkan passphrase. Ini memungkinkan login otomatis, tetapi kurang aman jika kunci privat dicuri.
    • Keamanan tambahan: Masukkan kata sandi yang kuat. Passphrase ini akan diminta setiap kali kunci digunakan, memberikan perlindungan ekstra terhadap penyalahgunaan kunci privat.

  4. Memverifikasi hasil:

    Periksa keberadaan berkas kunci dengan perintah berikut:

    ls ~/.ssh

    Output harus menunjukkan id_rsa dan id_rsa.pub.

Catatan penting: Kunci privat (id_rsa) harus disimpan dengan aman dan tidak boleh dibagikan. Kunci publik (id_rsa.pub) dapat disalin ke server.

Potensi masalah: Jika perintah gagal, pastikan direktori ~/.ssh memiliki izin yang benar (chmod 700 ~/.ssh).

Penyalinan Kunci Publik ke Server

Kunci publik harus disalin ke server agar autentikasi berbasis kunci dapat berfungsi. Cara termudah adalah menggunakan perintah ssh-copy-id.

Langkah-langkah:

  1. Menyalin kunci publik:

    Dari komputer klien, jalankan perintah berikut untuk menyalin kunci publik ke server.

    ssh-copy-id your_username@server_ip

    Ganti your_username dengan nama pengguna di server (misalnya, indodax) dan server_ip dengan alamat IP atau hostname server (misalnya, localhost atau 192.168.1.100).

  2. Memasukkan kata sandi:

    Masukkan kata sandi pengguna server saat diminta. Ini diperlukan sekali sebelum autentikasi kunci aktif.

  3. Memverifikasi akses:

    Uji login ke server untuk memastikan kunci berfungsi:

    ssh your_username@server_ip

    Jika passphrase dikosongkan, login akan otomatis. Jika passphrase diatur, Anda akan diminta memasukkannya.

  4. Menambahkan kunci untuk pengguna lain:

    Pengguna lain harus mengulangi langkah ini dari komputer mereka. Alternatifnya, tambahkan kunci publik mereka secara manual ke ~/.ssh/authorized_keys di server:

    sudo nano ~other_user/.ssh/authorized_keys

    Tambahkan isi id_rsa.pub pengguna lain, lalu atur izin:

    sudo chown other_user:other_user ~other_user/.ssh -R
sudo chmod 700 ~other_user/.ssh
sudo chmod 600 ~other_user/.ssh/authorized_keys

Potensi masalah: Jika ssh-copy-id gagal, pastikan login kata sandi masih diizinkan sementara di sshd_config (PasswordAuthentication yes) atau salin kunci secara manual.

Konfigurasi Server SSH untuk Autentikasi Kunci

Untuk meningkatkan keamanan, nonaktifkan login kata sandi dan wajibkan autentikasi berbasis kunci dengan mengedit berkas konfigurasi SSH.

Langkah-langkah:

  1. Mencadangkan berkas konfigurasi:

    Sebelum mengedit, buat cadangan berkas sshd_config untuk mencegah kehilangan konfigurasi.

    sudo cp /etc/ssh/sshd_config /etc/ssh/sshd_config.bak
  2. Mengedit berkas konfigurasi:

    Buka berkas sshd_config menggunakan editor teks seperti nano.

    sudo nano /etc/ssh/sshd_config
  3. Mengatur autentikasi kunci:

    Tambahkan atau ubah baris berikut di akhir berkas:

    PubkeyAuthentication yes
PasswordAuthentication no
ChallengeResponseAuthentication no
AllowUsers your_username

    Penjelasan:

    • PubkeyAuthentication yes: Mengaktifkan autentikasi berbasis kunci.
    • PasswordAuthentication no: Menonaktifkan login dengan kata sandi.
    • ChallengeResponseAuthentication no: Mencegah metode autentikasi alternatif.
    • AllowUsers your_username: (Opsional) Membatasi akses hanya untuk pengguna tertentu.

  4. Menyimpan dan memulai ulang layanan:

    Simpan berkas (Ctrl+O, Enter, Ctrl+X di nano), lalu mulai ulang layanan SSH.

    sudo systemctl restart sshd
  5. Memverifikasi konfigurasi:

    Uji login dengan kunci:

    ssh your_username@server_ip

    Uji login dengan kata sandi untuk memastikan gagal:

    ssh -o PubkeyAuthentication=no your_username@server_ip

    Hasil yang diharapkan: Login dengan kata sandi akan menghasilkan Permission denied.

Catatan penting: Jangan tutup sesi SSH aktif sebelum menguji login di sesi baru untuk menghindari kehilangan akses.

Pengaturan Firewall dan SELinux

Fedora menggunakan firewalld untuk pengelolaan firewall dan SELinux untuk kontrol keamanan tambahan. Pastikan keduanya dikonfigurasi dengan benar.

Firewall

  1. Membuka port SSH:

    Pastikan port 22 (default SSH) diizinkan oleh firewalld.

    sudo firewall-cmd --permanent --add-service=ssh
sudo firewall-cmd --reload

    Catatan: Opsi --permanent memastikan aturan tetap aktif setelah reboot.

  2. Memverifikasi aturan firewall:

    Periksa layanan aktif:

    sudo firewall-cmd --list-services

    Hasil yang diharapkan: Output harus mencantumkan ssh.

SELinux

  1. Menguji mode permisif:

    Jika login gagal, SELinux mungkin memblokir akses. Uji dengan mode permisif:

    sudo setenforce 0

    Coba login kembali. Jika berhasil, SELinux perlu penyesuaian.

  2. Mengatur mode permisif permanen:

    Edit berkas SELinux untuk perubahan permanen (opsional):

    sudo sed -i 's/SELINUX=enforcing/SELINUX=permissive/' /etc/selinux/config

    Alternatif: Untuk mode enforcing, buat aturan SELinux khusus menggunakan audit2allow berdasarkan log (sudo grep sshd /var/log/audit/audit.log).

Potensi masalah: Jika firewall memblokir, periksa zona aktif dengan sudo firewall-cmd --get-active-zones dan pastikan ssh diizinkan.

Pemecahan Masalah

Jika terjadi masalah selama pengaturan atau login SSH, ikuti langkah-langkah berikut untuk mendiagnosis dan menyelesaikannya.

Langkah-langkah:

  1. Memeriksa log SSH:

    Lihat log layanan untuk mengidentifikasi error.

    sudo journalctl -u sshd

    Contoh error: Authentication refused menunjukkan kunci salah atau pengaturan autentikasi tidak sesuai.

  2. Memeriksa izin berkas:

    Pastikan izin direktori dan berkas SSH benar:

    chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
  3. Memverifikasi kunci publik:

    Bandingkan isi ~/.ssh/authorized_keys di server dengan ~/.ssh/id_rsa.pub di klien.

  4. Menguji koneksi dengan mode verbose:

    Gunakan mode verbose untuk detail error:

    ssh -v your_username@server_ip

Catatan penting: Jika semua langkah gagal, pulihkan sshd_config dari cadangan (/etc/ssh/sshd_config.bak) dan uji ulang.
